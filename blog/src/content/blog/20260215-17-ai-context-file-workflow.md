AI와 긴 작업을 붙잡으면 대화 맥락은 금방 흐려진다. 처음엔 작업계획을 메모장에 따로 적었는데 코드와 분리돼 자주 어긋났다. 실제로 진행 상황을 물었을 때, 며칠 전에 만들고 갱신이 멈춘 md를 현재 상태처럼 가져온 적도 있었다. 그때부터 문제를 모델 성능보다 상태 관리로 봤다. 대화를 계속 잇는 방식은 카드집에 가깝고, 파일로 고정한 맥락이 있어야 세션이 바뀌어도 다시 붙는다.

이후에는 컨텍스트 파일을 에이전트가 직접 만들게 하고, 커밋할 때마다 최신화하도록 규칙을 걸었다. 파일에는 현재 스펙, 완료/진행/남은 작업, 다음 액션만 남겼다. 한 단위를 끝내면 컨텍스트를 비우고 새 세션에서 같은 파일로 다시 시작했다. 코드작성, 리뷰, 리팩토링, 문서갱신을 한 루프로 묶고, 테스트 선작성과 작은 단위 점진 진행을 같이 적용했다. 핵심은 긴 대화 유지가 아니라 짧은 주기의 재동기화였다.

체감상 가장 큰 변화는 복구 속도였다. 이전 결정의 이유를 다시 찾는 시간이 줄었고, 수동 메모를 따로 들고 다니는 부담도 줄었다. 반대로 파일이 한 번 stale 되면 모델은 그 문서를 사실로 취급했다. 그래서 갱신 시점을 커밋 이벤트에 묶는 게 중요했다. 또 여러 번 구현했다가 되돌리는 상황에서는 브랜치를 지우고 다음 세션에서 다시 시작하기도 했다. 비용은 들지만, 뒤늦은 후회를 메인 코드에 남기는 것보다는 낫다고 판단했다.

Git 위생 작업도 범위를 정해 위임했다. 메인 리베이스, 자잘한 커밋을 의미 단위로 묶기, 임시 브랜치 체리픽은 맡기고 최종 diff 판단은 사람이 했다. 이 방식이 만능이라는 뜻은 아니다. 정량 지표로 검증한 단계도 아직 아니다. 다만 장기 작업에서 중요한 건 한 번에 완벽히 쓰는 능력보다, 틀렸을 때 문서와 브랜치 상태를 빠르게 정리하고 다음 세션에서 다시 전진하는 운영 루틴이라는 점은 분명했다.

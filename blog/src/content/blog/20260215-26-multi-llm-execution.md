요즘 전달물 중심 개발을 하면서 한 모델만 고정하는 방식이 자주 막혔다. 클로드가 같은 맥락에서 헤매는 순간이 있었고, 제미나이가 그 지점을 풀어준 경우가 몇 번 있었다. 그래서 멀티 구성을 진지하게 보게 됐다. 다만 여기서 바로 부딪힌 건 기술보다 운영 제약이었다. 현재는 대부분 구독형으로 쓰고 있어 API 기반처럼 라우팅이나 자동 폴백을 쉽게 묶기 어렵다. 모델 성능만 보고 결정할 수 없는 이유가 분명해졌다.

프롬프트가 잘 안 먹히는 날에는 지시를 길게 늘리기보다, 잘 되던 날의 예시를 few-shot으로 붙여 재현성을 먼저 맞췄다. 이 방식이 추상적인 “잘 해줘”보다 훨씬 안정적이었다. 실행 쪽에서는 로드맵을 기준으로 마일스톤을 잡고, 개발 이슈를 생성해 이슈보드에 등록하는 흐름을 만들었다. 초안 작성부터 팀별 배포 문서까지 한 번에 뽑아내는 작업도 같은 맥락에서 처리했다. 손으로 시작할 때보다 출발 속도는 확실히 빨라졌다.

결과가 좋았던 만큼 한계도 바로 드러났다. 생성 속도가 올라가자 병목은 작성이 아니라 검토로 이동했다. 문서와 이슈가 빠르게 쌓이는데, 사람이 읽고 결정하는 속도는 그만큼 늘지 않았다. 그래서 “많이 만드는 것”보다 “무엇을 바로 쓰고 무엇을 다시 검토할지”를 먼저 정하는 기준이 필요했다. 리뷰 부담을 줄이지 못하면 자동화가 오히려 팀 컨텍스트를 흐리게 만들 수 있다.

지금 기준에서 적용 포인트는 단순하다. 모델은 우열보다 실패 유형 기준으로 분리해 두고, 프롬프트는 잘 된 사례를 few-shot으로 고정해 흔들림을 줄인다. 로드맵-마일스톤-이슈보드 자동화는 생성 설계보다 검토 설계를 먼저 둔다. 코딩 보조 도구 의존이 커진 현실도 부정하지 않되, 판단과 마감 기준만큼은 사람이 끝까지 쥐는 편이 안전했다. 정량 지표를 크게 쌓은 단계는 아니라 단정은 어렵지만, 막히는 구간을 줄이는 데는 이 조합이 실무에서 충분히 유효했다.

팀장 맡고 첫 주에 신규 기능보다 구조를 먼저 손봤다. 백엔드 인원보다 서비스 수가 많고 레포도 쪼개져 있었다. 서비스 간 HTTP 호출은 길을 잃고 타임아웃이 났고, 비동기 작업은 메시지 브로커에 들어간 뒤 로그 없이 사라지기도 했다. API 명세가 바뀌면 로그인 실패나 상품 이용 불가가 바로 사용자 문제로 번졌다. Datadog을 붙여도 트레이스가 구간마다 끊겨 원인 찾는 시간만 늘어났다.

이 상황에서 스케일링 계획을 더 얹는 건 해결이 아니라 부채 증설에 가깝다고 봤다. 그래서 새 도구 추가보다 삭제를 먼저 택했다. 여러 레포를 합치고, 마이크로서비스처럼 나뉜 경계를 다시 묶고, 메시지 브로커 제거를 시작했다. 핵심은 ‘관심사 분리’라는 구호가 아니라 팀 크기와 운영 현실에 맞는 경계였다. 다행히 기존 구성원도 같은 필요를 느끼고 있어 방향 합의는 빨랐다.

비슷하게 미래 가능성을 이유로 미리 넣는 설계도 다시 봤다. 언어가 추가될지 모른다는 i18n, DB를 바꿀 수 있다는 전제의 과한 추상화, 언젠가 플랫폼을 옮길 수 있다는 준비 코드가 실제로는 오래 남아 유지비만 만드는 경우가 많았다. 냄새가 나는데 방향제를 뿌리듯 인덱스 추가나 브라우저 프리렌더링 같은 처방을 바로 얹으면, 근본 문제 위에 새 문제가 겹치기 쉽다.

삭제 우선 접근은 멋진 방법론이 아니라 실패 지점을 줄이는 운영 전략에 가깝다. 호출 경로를 짧게 만들고 책임 경계를 단순화하면 장애가 나도 추적할 면이 줄고, 팀이 같은 그림으로 의사결정하기 쉬워진다. 물론 이 방식이 영구 해법은 아니다. 트래픽과 조직 복잡도가 실제로 커지면 다시 분리와 최적화가 필요해진다. 내 기준은 단순했다. 할 수 있는 단순화와 제거를 먼저 끝내고, 그래도 부족할 때만 다음 복잡도를 도입한다.

에이전트 코딩을 쓰면 바로 빨라질 거라 기대했지만, 초반에 먼저 보인 건 생산성보다 소모였다. 구독 시작 당일에도 오후 2시에 리셋된 토큰이 퇴근 무렵 거의 바닥났다. 같은 문제를 오래 붙잡을수록 품질이 선형으로 오르기보다, 맥락이 흐려진 상태에서 검토 비용만 늘어나는 구간이 빨리 온다는 걸 체감했다.

프론트 테스트 이슈가 그 패턴을 분명히 보여줬다. 의존성 문제를 버전 충돌로 잘못 해석한 뒤 수정을 반복하면서 토큰을 크게 썼고, 세션을 세 번 다시 열어도 결론은 비슷했다. 결국 사람 디버깅으로 임포트 순서를 조정해 해결했다. 이 뒤로는 다섯 번 안에 의도가 맞지 않으면 `/clear` 하고 다시 시작한다. 길게 설득하는 것보다 빠른 리셋이 비용을 줄였다.

토큰이 빨리 닳는 이유는 모델 성능보다 작업 방식에 가까웠다. 같은 설명을 다시 시키고, 결과를 또 검토시키고, 문서를 계속 고치게 하면 결과는 조금 나아져도 소모는 급격히 커진다. 그래서 세션 끝에는 작업 중 파악한 내용과 스펙을 문서로 남기고, 브랜치를 정리한 뒤 다음 세션을 새로 연다. 또, 맥락 문서 없이 일부 구현만 툭 던지는 요청은 사람이든 LLM이든 실패 확률이 높아서, 시작 전에 목적지와 접근 방법을 먼저 맞춘다.

팀 핸즈온에서도 차이는 명확했다. 실무 이슈를 작은 단위로 쪼개 계획을 관리했을 때 프론트 대상 세션은 잘 진행됐지만, 백엔드는 API 통합 테스트 환경을 처음 잡는 구간에서 막히며 토큰만 많이 썼다. 결론은 단순했다. 에이전트 코딩은 코드 생성보다 환경 의존 구간에서 더 자주 무너진다. 아직 이 방식의 장기 생산성을 수치로 확정하진 못했지만, 실패를 빨리 끊고 재시작 가능한 상태를 유지하는 데는 확실히 효과가 있었다.

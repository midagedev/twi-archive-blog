시작점은 단순했다. 팀은 코파일럿도 쓰지 않았고 ChatGPT 정도만 쓰는 상태였다. 에이전트 활용은 사실상 없는 편이었다. 그래서 개인 구독으로 에이전트를 붙여, 한 달 안에 눈에 보이는 결과를 만들겠다는 목표부터 세웠다. 핵심은 내가 빠르게 코드를 치는 게 아니라, 반복 업무를 누가 해도 비슷하게 처리되게 만드는 것이었다.

처음에는 귀찮은 일을 매번 프롬프트로 처리했는데 곧 한계를 봤다. 같은 설명을 계속 붙여야 했고, 환경 차이 때문에 결과가 흔들렸다. 그래서 반복되는 작업을 클로드코드 스킬로 쪼개서 고정했다. 깃허브 전용 흐름이 아닌 환경이라 PR 관련 작업은 따로 만들었고, 스킬 안에 우리 팀 업무 컨벤션을 넣어 실행 편차를 줄였다. 급한 상황에서는 원격으로 지시만 넘겨 배포까지 이어지게 만든 것도 이 흐름의 연장선이었다.

테스트 쪽에서는 기존에 만든 TC가 뒤늦게 가치를 냈다. 팀원이 커밋 변경점에서 관련 TC를 추정해 선택 실행하는 작업을 만들었고, 에이전트가 코드베이스를 넓게 뒤져 후보를 잡아줬다. 전체 테스트를 매번 돌리는 방식보다 대기와 소음이 줄어들었고, 변경과 무관한 실패에 끌려가는 시간도 덜했다. 정량 수치는 아직 충분히 쌓지 못했지만, 체감상 정확도와 실용성은 분명했다.

Playwright MCP를 붙인 뒤에는 브라우저 단 분석이 실무적으로 바뀌었다. 파일 업로드 후 전송 버튼을 눌렀을 때 발생한 XHR 요청을 모아, 어떤 API가 어떤 쿼리로 호출됐는지 바로 정리할 수 있었다. 여기에 서버 코드를 함께 분석시키면 화면 동작과 백엔드 처리의 연결을 빠르게 확인할 수 있다. e2e 작성과 API 추적을 따로 하지 않아도 되는 점이 컸다.

깃 이력 조사도 방식이 달라졌다. 예전처럼 블레임을 오래 보는 대신, 언제부터 왜 이렇게 됐는지 먼저 조사시키고 필요한 수정 흐름을 이어간다. 히스토리 탐색, 코드 초안, 커밋/PR, 충돌 해결 같은 보조 작업을 한 줄기로 묶을 수 있었다. 다만 최종 판단과 책임은 여전히 사람의 몫이고, 스킬에 규칙을 명확히 넣지 않으면 품질이 흔들린다. 적용 포인트는 단순하다. 잘 쓰는 프롬프트를 늘리는 데서 멈추지 말고, 반복되는 결정을 팀 자산으로 고정하는 것.

팀장을 맡고 첫 일주일 동안 기능 추가보다 먼저 구조를 줄이는 일을 시작했다. 레포를 나눠 둔 상태와 마이크로서비스처럼 쪼개 둔 구성이 팀 규모에 비해 과했다. 백엔드 인원보다 서비스 수가 더 많은 상황에서는 관심사 분리가 장점보다 운영 부담으로 먼저 돌아왔다. 다행히 기존 구성원들도 같은 필요성을 이미 느끼고 있었다.

문제는 장애 양상에서 바로 드러났다. 서비스 간 HTTP 호출에서 요청이 길을 잃고 타임아웃이 자주 났고, 메시지브로커로 넘긴 비동기 작업은 로그도 없이 사라지는 경우가 있었다. API 명세가 달라진 여파로 로그인 실패가 생기거나 상품 이용이 막히기도 했다. Datadog을 써도 트레이스가 중간에 끊겨 원인 추적이 자주 늦어졌다.

그래서 이번 주에 잡은 방향은 단순했다. 모든 레포를 합치고, 과하게 나뉜 서비스를 다시 합치고, 메시지브로커를 없애는 작업을 시작했다. 경계를 늘려 얻는 이점보다, 경계를 관리하는 비용이 더 큰 상태였기 때문이다. 작은 서비스마다 서버를 하나씩 붙이는 방식은 기술 복잡도뿐 아니라 클라우드 비용도 키웠다.

아직 일주일이라 결과를 단정할 단계는 아니다. 다만 지금 맥락에서는 더 많은 도구보다 더 짧은 호출 경로와 일관된 명세, 끊기지 않는 로그가 우선이다. 팀 크기와 운영 역량을 기준으로 구조를 정하지 않으면, 안정성을 위해 도입한 복잡성이 오히려 안정성을 깎는다는 점은 분명했다.

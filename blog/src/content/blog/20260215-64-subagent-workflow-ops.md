메인 세션과 서브 에이전트를 나눌 때 처음부터 공유 메커니즘이나 실시간 동기화를 붙이고 싶어질 때가 많다. 하지만 운영 관점에서는 그 단계가 가장 비용이 크고, 실패 원인도 흐려지기 쉽다. 그래서 먼저 고정한 원칙은 단순했다. 계획은 메인 세션에서 만들고, 서브 에이전트에는 실행 가능한 단위만 넘긴다. 역할을 분리하면 어디서 문제가 생겼는지 빠르게 드러나고, 수정도 작은 단위로 끝낼 수 있다.

실행 구조도 같은 기준으로 가져갔다. 서브 에이전트끼리 상태를 공유하게 만들기보다, 메인 세션이 작업 순서와 기대 결과를 정리해 전달했다. 또 내가 원격에서 직접 모든 세션을 돌리기보다, 팀원이 로컬에서 실행하도록 두었다. 대신 로컬에 남는 세션 파일을 분석해 지시가 흔들린 구간, 반복이 생긴 구간, 불필요한 왕복이 생긴 구간을 확인했다. 복잡한 제어보다 기록 기반 점검이 먼저였다.

가시성은 티켓 보드만으로는 충분하지 않았다. 지라에 드러나지 않는 활동을 보려면 채팅과 문서 흐름까지 같이 봐야 한다. 그래서 슬랙(우리 팀은 팀즈) 대화와 컨플루언스 같은 문서도 주기적으로 분석 대상에 넣었다. 프롬프트 문장만 평가하지 않고 작업 흐름 자체를 리뷰 대상으로 둔 이유도 이 지점이다. 완전한 실시간 파이프라인은 아니어도, 운영 감각을 끌어올리기에는 이 구성이 충분히 실용적이었다.

결국 핵심은 확장 순서였다. 먼저 단순한 구성으로 가치가 나는 케이스를 몇 번 만들고, 병목이 반복될 때만 공유 구조를 붙이는 편이 안정적이었다. 공개된 정보에는 팀 규모나 처리량 같은 정량 지표가 없어 여기서는 의사결정과 실행 방식만 정리했다. 적용 포인트는 분명하다. 메인 세션의 계획 책임을 고정하고, 서브 에이전트의 출력 계약을 명확히 하고, 로컬 아티팩트 분석으로 가시성을 확보한 뒤에 다음 복잡도를 올리는 순서가 좋다.

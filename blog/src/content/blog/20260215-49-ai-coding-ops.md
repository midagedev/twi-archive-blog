주변에서 AI를 어떻게 그렇게 쓰냐고 물으면, 특별한 비법은 없다고 말한다. 내가 반복하는 건 단순하다. 모델이 내 요청을 어떻게 이해했는지 빠르게 확인하고, 어긋난 지점을 짧은 이터레이션으로 계속 맞춘다. 처음부터 완성된 도착지를 못 그려도 괜찮다. 대신 중간 산출물을 봤을 때 “이 방향이 맞는가”를 판단할 기준은 사람이 쥐고 있어야 한다.

기준이 없으면 모델은 가장 흔한 쪽으로 미끄러진다. CI용 YAML을 요청했는데 Azure DevOps가 아니라 GitHub 기준으로 나오는 일이 그랬다. 이때는 모델 탓보다 입력 컨텍스트를 먼저 본다. 웹 검색을 붙여 실제 Azure 사용례를 다시 찾아오게 하니 결과가 금방 교정됐다. 이후에는 요청문에 플랫폼, 실행 환경, 피해야 할 문법을 처음부터 명시하는 습관을 굳혔다.

또 자주 마주치는 실패는 그럴듯한 오답이다. 질문을 반대로 던져도 모델은 그 조건에 맞춰 이유를 만들어낸다. 그래서 답변을 “정답”이 아니라 “가설”로 취급한다. 핵심 전제를 한 줄씩 다시 말하게 하고, 반대 질문으로 모순이 생기는지 확인한다. 이 검증 루프를 건너뛰면 당장은 빨라도, 뒤에서 디버깅 비용이 더 크게 돌아온다.

코드 결과물은 의도적으로 화려함을 낮춘다. 멋진 추상화보다 팀이 바로 읽고 고칠 수 있는 평이한 코드가 필요한 순간이 더 많았다. 그래서 종종 세련된 해법 대신 단순하고 싼 구현을 주문한다. 프롬프트 가공에 구현 시간보다 더 오래 쓰는 날도 있지만, 변경 단위를 작게 유지하면 리뷰와 롤백이 쉬워진다. 다만 이 방식의 효과를 정량으로 얼마나 얻는지는 아직 충분히 측정하지 못했다.

한계도 분명하다. 에이전트 작업에 익숙해진 뒤 오랜만에 손으로 같이 코딩하면, 머릿속 패턴은 있는데 손이 바로 따라오지 않을 때가 있었다. 그래서 지금은 AI를 계속 쓰되, 일부 시간은 직접 타이핑과 기본기 복원에 따로 배정한다. 내 운영 원칙은 단순하다. AI는 작성자 역할을 맡고, 사람은 방향과 품질을 관리하는 운영자 역할을 끝까지 놓지 않는다.

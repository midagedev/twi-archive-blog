자바, 파이썬, 자바스크립트, 타입스크립트는 계속 VSCode 하나로 처리해 왔다. 그래서 처음 코틀린 코드를 만졌을 때도 먼저 든 생각은 단순했다. 젯브레인 언어인데 IntelliJ를 써야 하나, Xcode 없이 Swift를 만지는 것과 비슷한 무리수인가 하는 의문이었다.

막상 손을 대보니 핵심 변수는 IDE가 아니었다. 테스트와 빌드는 Gradle이 처리했고, 코드 작성은 Claude 도움을 받아 진행했다. 이 조합에서는 에디터를 바꾸는 비용보다 기존 흐름을 유지하는 이점이 더 컸다. 결국 “코틀린이면 IntelliJ” 같은 고정 규칙보다, 지금 해야 하는 작업 단위가 무엇인지가 더 실용적인 기준이 됐다.

비슷한 패턴은 Swift에서도 있었다. 수정이 필요할 때 Xcode와 VSCode를 함께 띄워두고, 실제 편집은 Copilot을 쓰기 편한 VSCode에서 진행한 적이 많았다. 플랫폼 전용 IDE가 필요한 구간은 분명 있지만, 코드 작성과 반복 수정만 놓고 보면 중심 도구가 조금씩 이동하고 있다는 감각이 있었다.

정리하면 특정 IDE를 부정하려는 이야기는 아니다. 이 스레드에서 확인된 사실은, 이번 코틀린 첫 작업에서는 빌드 체인과 보조 도구가 우선이었고 IDE 선택 우선순위는 상대적으로 낮았다는 점이다. 다만 디버깅이나 IDE 전용 기능이 필요한 상황은 여기서 다루지 못했으니, 그 구간에서는 판단이 달라질 수 있다.

팀장을 맡고 첫 주에 먼저 한 일은 새 기능 설계가 아니라 구조 진단이었다. 레포는 여러 개로 쪼개져 있었고 서비스 수는 백엔드 인원보다 많았다. 관심사 분리라는 명분은 있었지만, 실제 팀 규모와 운영 역량에 비해 경계가 과했다. 이 상태에서는 코드 완성도보다 전달 경로와 조정 비용이 먼저 장애를 만든다.

증상은 이미 운영에서 반복되고 있었다. 서비스 간 HTTP 호출이 중간에서 길을 잃어 타임아웃이 나고, 메시지 브로커로 넘긴 비동기 작업이 로그 없이 사라지는 경우가 있었다. API 명세가 바뀐 뒤 연동 지점이 따라오지 않아 로그인 실패나 상품 이용 불가도 생겼다. Datadog을 써도 트레이스가 자주 끊겨 원인 추적 시간이 길어졌다. 관측 도구를 더 붙이는 것만으로는 해결되지 않는 종류의 문제였다.

그래서 방향을 반대로 잡았다. 분리를 더 정교하게 만드는 대신 레포를 합치고, 마이크로서비스처럼 나뉜 경계를 통합하고, 메시지 브로커 제거를 시작했다. 목표는 기술 스택을 화려하게 만드는 게 아니라 실패 지점을 줄이는 것이었다. 호출 단계를 줄이고 변경 책임을 한곳으로 모으면, 장애 확인과 수정 배포가 같은 흐름에서 끝난다. 작은 팀일수록 이 단순함이 품질과 속도를 같이 지킨다.

다만 기술 선택만으로 끝나지는 않았다. 2026년 1월에는 인프라팀이 이미 구성한 쿠버네티스를 유지하는 쪽으로 결론이 나면서 배포에 결재 절차가 붙었다. 단순화를 시도해도 조직 경계와 의사결정 구조가 복잡하면 운영 비용은 다시 올라간다. 트윗에 공개된 정보만으로는 비용 절감률이나 장애 감소 폭 같은 정량 수치는 확인할 수 없다.

이 경험 이후 기준은 분명해졌다. 팀 규모가 작고 장애의 상당수가 서비스 경계에서 나온다면, 경계를 지키는 일보다 경계를 줄이는 일이 더 품질에 가깝다. 데브옵스 역량도 복잡한 도구를 많이 다루는 능력만으로는 부족하고, 정말 복잡해야 하는지 먼저 묻는 판단이 필요하다. 아키텍처는 유행이나 신념이 아니라, 지금 팀이 안정적으로 운영할 수 있는 단위로 선택해야 한다.

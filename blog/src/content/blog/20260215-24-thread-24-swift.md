집에서 개인 프로토타입을 다시 잡으면서 시작점을 Swift로 정했다. 이유는 단순했다. OS가 제공하는 음성 인식이 예상보다 빠르고 정확했고, 웨이크워드도 구현 난도가 높지 않았다. OpenAI Realtime Audio API가 더 강력한 선택지인 건 맞지만, 지금 단계에서는 비용이 먼저 걸렸다. 그래서 LLM API 모드와 오디오 API 모드를 같이 붙여 두고, 상황에 따라 경로를 바꿔볼 수 있게 구성했다.

초기 사용성 확인은 집에서 바로 했다. 이미지 생성 API를 연결하고, 결과를 출력까지 이어지는 흐름을 빌트인 툴로 묶어 두니 아이들이 실제로 계속 사용했다. 이틀 동안 괴물 이미지 170장을 만들었고, 이 사용량에서도 비용은 천 원이 안 들었다. 숫자가 크다는 의미보다, 기능을 붙였을 때 바로 반복 사용이 나왔다는 점이 더 중요했다.

다만 테스트가 쉽지는 않았다. 아이를 테스터로 두면 예상한 경로보다 즉흥적인 입력이 훨씬 많아서, 막히는 지점이 빨리 드러난다. 그 과정을 불편으로만 보지 않고 다음 수정 우선순위를 정하는 신호로 썼다. 구현 자체는 재미있었지만, 재미와 별개로 반복 가능한 개발 리듬이 필요했다.

그래서 로드맵 기준으로 마일스톤을 먼저 쪼개고, 개발 이슈를 생성해 보드에 등록하는 흐름까지 붙였다. 에이전트를 팀처럼 나눠 운영하는 방식이 특히 유효했다. 아직 정확한 품질 지표나 장기 운영 데이터는 없다. 대신 현재까지는 속도, 비용, 실제 사용 반응을 한 번에 확인할 수 있는 구조를 만들었고, 다음 단계로 넘어갈 최소 조건은 확보했다.

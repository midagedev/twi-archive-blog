LLM이 들어오면서 코드 작성 속도는 빨라졌지만, 팀 품질이 자동으로 좋아지지는 않았다. 오히려 병목은 리뷰로 몰렸다. 그래서 나는 먼저 “얼마나 빨리 짜느냐”보다 “얼마나 빨리, 정확히 검토하느냐”를 기준으로 두려고 했다. 실제로 LLM 시대에는 리뷰 역량이 더 중요하다고 보고, 코드리뷰 방식 자체를 점검하는 컨설팅까지 제안했다. 다만 이 부분은 내가 끝까지 챙기지 못해 아직 진행 중인 과제로 남아 있다.

실무에서 바로 적용한 원칙은 단순했다. 변경이 있으면 그 변경을 다루는 테스트 코드를 PR에 같이 넣는다. 그러면 작동 여부는 테스트로 판단하고, 리뷰는 코드 품질 중심으로 가져가기 쉬워진다. 리뷰어가 매번 수동 재현에 시간을 쓰지 않아도 되고, 대화도 “되나/안 되나”에서 “이 구현이 유지 가능한가”로 이동한다. 생성 속도가 빨라질수록 이 분리는 더 중요했다.

브랜치도 같은 맥락으로 봤다. 목적은 충돌 방지 자체가 아니라, 브랜치를 출발점으로 PR과 코드리뷰까지 이어지는 흐름을 만드는 데 있다. 이 고리가 없으면 변경 이력이 흩어지고 피드백도 기록으로 남지 않는다. GitHub를 쓰면 커밋 코멘트가 편해 운영하기 좋지만, 꼭 GitHub여야 하는 것은 아니다. git CLI나 IDE 지원만으로도 같은 흐름은 만들 수 있다. 핵심은 도구 이름보다 리뷰로 연결되는 기본 경로를 팀 표준으로 두는 일이다.

운영 디테일에서는 또 다른 한계도 보였다. QA 보드 이름, 팀 보드 구분, 계정 식별, 브랜치 시작점 같은 항목은 통합 스크립트로 묶으려면 생각보다 정밀한 코드가 필요했다. 그래서 우선은 MD에 절차를 말로 정리하고, 필요한 단위 스크립트로 나눠 쓰는 방식을 택했다. LLM은 이런 문서를 실행 단위로 옮기는 데 유용하지만, 어떤 순서로 검토하고 승인할지는 결국 팀 규칙이 정해야 한다. 속도가 올라간 팀일수록 리뷰 중심 설계를 먼저 두는 이유가 여기에 있다.

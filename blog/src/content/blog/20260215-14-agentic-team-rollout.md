회귀 이슈가 QA와 개발팀 둘 다를 지치게 하던 시점에, QA팀에서 받은 TC 3천 개를 사람 손으로 순차 처리하는 방식부터 접었다. 개인 프롬프트를 잘 쓰는 사람 몇 명에 기대서는 처리량이 나오지 않았다. 그래서 클로드코드 스킬로 케이스 그룹 선별·분석, Playwright 통과 테스트 작성, PR 생성까지 이어지는 고정 루프를 먼저 만들었다. 포인트는 자동화 자체보다, 팀 누구나 같은 순서로 반복할 수 있는 단위를 정하는 데 있었다.

이걸 개인 생산성 실험으로 두지 않고 팀 운영 규칙으로 옮겼다. 팀원 전원이 같은 루프를 돌리게 하고, 주니어와는 주 2시간 정도 코드 품질 대화를 따로 잡았다. 예전에는 아쉬운 리팩토링 PR을 매몰비용이나 감정 때문에 리젝하기 어려운 순간이 있었는데, 지금은 피드백 후 에이전트 재작성으로 바로 다시 검토할 수 있다. 논쟁보다 반복이 쉬워지면서 collective code ownership을 실제 습관으로 붙이는 데 도움이 됐다.

도입 장벽을 낮추기 위해 사용 패턴도 정리했다. 에이전트 사용이 낯선 팀원을 위해 4개 세션 기록을 jsonl로 분석해 막히는 패턴을 추렸고, 새 합류 인원과는 페어 프롬프팅으로 환경부터 맞췄다. 구버전 도구나 기본 모델에 묶여 있으면 같은 요청도 결과 편차가 컸다. 터미널 팬을 나눠 병렬로 작업하고, 오래된 진행 md를 최신으로 갱신하게 하니 에이전트가 현재 작업 상태를 잘못 읽는 문제도 줄었다.

현재는 하루 약 300개씩 테스트 케이스가 PR 단위로 배출되고 있다. 다만 이 수치가 곧 품질 완료를 뜻하지는 않는다. 회귀를 완전히 없앴다고 말하기는 아직 이르고, 문서 갱신이 늦어지면 루프가 다시 흔들린다. 그래도 팀 단위에서 확인한 건 분명하다. 좋은 프롬프트 한 줄보다, 공통 루프와 최신 맥락 문서, 짧고 자주 하는 코칭을 함께 돌릴 때 처리량과 일관성이 같이 올라간다.

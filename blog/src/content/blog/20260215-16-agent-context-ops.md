에이전트가 강해질수록 역설적으로 컨텍스트 운영이 더 중요해졌다. 대화를 길게 이어가면 처음엔 빠르게 느껴지지만, 어느 순간부터 근거가 흐려졌다. 실제로 작업 현황을 물어봤을 때 며칠 전에 만들어 두고 갱신하지 않은 md 파일을 현재 상태처럼 가져온 적이 있었다. 새 프레임워크처럼 학습량이 적은 영역은 더 불안정했고, 공식 문서를 넣어도 생성 품질 편차가 컸다. 문제는 모델 성능만이 아니라, 내가 작업 상태를 고정해 두는 장치를 갖고 있지 않았다는 데 있었다.

그래서 메모장 따로 관리하던 계획을 버리고, 저장소 안에 작업 계획 파일을 두고 에이전트에게 직접 생성·갱신시키는 방식으로 바꿨다. 기준은 단순했다. 의미 있는 커밋이 생길 때마다 완료 항목, 남은 작업, 다음 액션을 같은 파일에 반영한다. 작업은 테스트, 작은 단위 구현, 리뷰, 리팩토링, 문서 갱신 순서로 짧게 끊고, 세션이 길어지면 컨텍스트를 비우고 다시 시작했다. 다음 세션은 이전 대화가 아니라 이 파일을 시작점으로 삼았다.

체감 차이는 컸다. 대화만 누적할 때가 카드로 만든 집이었다면, 파일 기준으로 돌린 뒤에는 레고에 가까웠다. 흐름이 끊겨도 현재 위치를 복구하기 쉬웠고, 브랜치 정리나 커밋 재구성 같은 운영성 작업도 덜 흔들렸다. 다만 이 방식도 자동으로 유지되지는 않는다. 파일이 밀리면 에이전트는 자신 있게 오래된 맥락을 재사용한다. 정량 지표를 따로 남기지 않아 속도 향상을 수치로 말하긴 어렵지만, 잘못된 방향으로 길게 가는 횟수는 줄었다.

지금은 규칙을 세 가지로 고정했다. 계획 파일은 코드와 같은 브랜치에 둔다, 커밋 단위로 반드시 갱신한다, 그리고 주기적으로 세션을 리셋한다. 여기에 “지금까지 파악한 스펙과 남은 리스크를 문서에 다시 써라”를 루프마다 붙인다. 이 습관이 있으면 완성 직전에 전부 지우고 다시 쓰는 상황을 완전히 없애진 못해도, 왜 다시 쓰는지와 어디서 어긋났는지는 남는다. 결국 에이전트 활용의 핵심은 한 번의 정답보다 상태를 잃지 않는 운영에 가깝다고 본다.

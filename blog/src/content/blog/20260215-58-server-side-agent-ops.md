처음에는 pod에 간단한 이미지를 띄우는 정도라 큰 문제가 없었다. 그런데 코딩 에이전트를 서버에서 본격적으로 돌리기 시작하자 바로 한계가 드러났다. 가장 먼저 막힌 건 메모리였고, 기본 스펙으로는 작업이 빡빡해서 4GB로 올렸다. 이때 배운 건 기능 구현보다 실행 환경의 최소 조건을 먼저 정해야 한다는 점이었다.

실행 경로도 로컬 중심보다 서버 중심이 효율적이었다. 서버에서 에이전트를 켜면 GitOps용 레포를 만들고 원격에 올린 뒤, 필요한 YAML을 바로 ArgoCD에 반영하는 흐름이 짧아진다. 체감상 편의성은 분명히 커졌다. 대신 자동화가 배포 권한과 맞닿기 때문에, 어디까지 에이전트에 맡기고 어디부터 사람이 확인할지 경계를 먼저 정해 두는 게 필요했다.

운영 규모를 넓히면 정책 리스크가 같이 따라온다. 실제로는 토큰이 부족한 사람 인증을 도와주거나, EC2 세 대에서 논인터랙티브 모드로 돌리는 방식까지 써봤다. 이 정도가 곧바로 심한 위반처럼 느껴지진 않았지만, 납득하기 어려운 밴 사례가 보인 것도 사실이었다. 기준이 명확히 공개되지 않은 영역이라, 이 부분은 기술 문제가 아니라 불확실성을 전제로 한 운영 규칙 문제로 봐야 했다.

병렬 실행도 단순히 개수를 늘리는 방식은 오래 못 간다. 주의력이 필요한 메인 작업 한두 개와, 상대적으로 가벼운 작업 두세 개를 분리해 다루는 방식이 더 안정적이었다. 탭과 팬 배치도 그 기준에 맞춰 나눴고, Max20 요금제에서는 그 이상 확장하기 어려웠다. 요금제는 이름보다 실제 사용량 대비 단가로 봐야 하고, 상위 플랜이 Max5 대비 사용량 4배에 가격 2배 수준이라는 설명이나 일반 업무시간 기준 100달러 플랜 체감처럼, 결국 팀의 사용 패턴으로 역산해 정하는 게 맞다.

처음 보는 코드베이스에 들어갈 때 가장 오래 걸리는 건 기능 구현보다 맥락 회수였다. 클로드 코드를 처음 써본 날에는 이 구간이 확실히 줄었다. 이전에 쓰던 도구보다 시작이 덜 끊기고, 탐색에서 수정으로 넘어가는 흐름이 매끄러웠다. 체감상 차이는 클라이언트와 모델이 붙어 있다는 점인데, 결과적으로 작업 전환 비용이 줄어든 쪽에 가까웠다.

그래서 온보딩을 넘어서 리뷰 자동화도 시도했다. 로컬 PC에 웹훅을 열고, 구독 계정이 붙은 환경에서 레포를 클론한 뒤 브랜치를 체크아웃해서 리뷰를 돌리는 방식이었다. 다만 여기서부터는 디테일이 성패를 갈랐다. QA 보드 이름, 팀 보드 구분, 계정 문맥, 브랜치 기준점 같은 값은 통 스크립트로 한 번에 묶으면 금방 깨졌다. 반대로 MD로 규칙을 풀어두고 단위 스크립트로 나누게 하니 세팅이 훨씬 덜 막혔다.

한계도 바로 드러났다. 스프링은 패턴이 정형화돼 있으니 캐시 붙이기는 쉬울 거라고 봤지만, 인메모리 캐시와 레디스를 같이 쓰는 테스트에서 빈 주입 오류를 끝내 정리하지 못했다. 이 지점은 모델이 코드를 생성하는 능력보다 실행 환경의 조건과 테스트 컨텍스트 이해가 더 중요했다. 자동 리뷰 시도 역시 아직 세팅 단계라, 실제 운영에서의 정확도나 비용 효율은 지금 시점에 단정하기 어렵다.

그래서 요즘은 경계를 나눠서 쓴다. 쿼리성 MCP처럼 컨텍스트를 크게 먹는 작업은 메인 세션에 바로 넣지 않고, 서브 세션에서 처리한 요약만 가져오는 쪽을 우선 고려한다. 메인 세션은 의사결정과 변경 기록에 집중하고, 장기 컨텍스트는 파일시스템으로 공유한다. 정리하면 LLM 코딩은 온보딩과 반복 작업 정리에 강하고, 환경 의존성이 높은 통합 구간에서는 사람이 끝까지 책임지는 구조가 아직 가장 안정적이다.

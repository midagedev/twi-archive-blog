백엔드팀이 만들어진 지 2개월쯤 됐을 때, 백오피스 개편 일정이 잡히면서 역할 경계가 다시 흐려졌다. 프론트팀은 Next.js 중심으로 일하고 있어서 기존 Svelte 백오피스는 가능하면 건드리지 않으려 했고, 결국 백엔드팀이 BFF뿐 아니라 화면 영역까지 같이 보는 구조로 돌아왔다. 작은 조직에서는 역할 정의보다 로드맵 압력이 먼저 들어오고, 그때부터 직무 구분은 문서보다 일정표에 의해 다시 그려진다. 그래서 누가 맞는 담당자인지 따지기보다, 어디를 먼저 고정해야 전체가 덜 흔들리는지부터 판단하게 됐다.

내가 먼저 손댄 건 API 경계였다. 프론트에서 쿼리스트링을 만드는 방식은 라이브러리마다 차이가 있는데, 백엔드 파서가 콤마 분리나 중복 키를 리스트로 받아주면 호출 코드가 단순해진다. 이건 프론트 편의만의 문제가 아니라, 팀이 분산되어도 계약을 유지할 수 있게 만드는 운영 장치에 가깝다. Next.js와 SvelteKit처럼 화면 기술이 달라도, 입력 형태와 응답 규칙이 안정되면 기능 변경 때 마찰이 줄었다.

실행 단계에서는 언어나 레이어를 가리지 않고 필요한 곳을 직접 수정했다. Python, Go, Next.js, SvelteKit 코드를 오가고, 데브옵스 도움을 받아 Kubernetes YAML을 수정해 배포도 진행했다. Lens로 클러스터 상태를 보면서 설정을 확인하고, ArgoCD에서 배포별로 실패를 미루고 성공한 버전을 promote해보니 어디서 병목이 생기는지 훨씬 빨리 보였다. 스택의 일관성을 완벽하게 맞추는 것보다, 실패 지점을 빨리 드러내는 흐름을 먼저 확보하는 게 당시에는 더 실용적이었다.

이후 모노레포에서 NestJS와 React를 함께 운영하고 외부 API 제공까지 고려하니, 타입 공유와 서버 호출 방식 선택이 다시 핵심 이슈가 됐다. tRPC는 매력적이지만 팀 합의 없이 밀면 개인 취향으로 보일 수 있고, ts-rest는 계약 중심으로 설명하기 쉬워서 합의 비용이 낮다. 결국 중요한 건 도구 이름이 아니라 온보딩 난이도, 외부 API 호환성, 배포 독립성 같은 기준을 먼저 맞추는 일이다. 팀 규모나 릴리스 주기 같은 세부 맥락은 공개된 정보가 제한적이라, 최종 선택은 각 팀의 제약에 이 기준을 대입해 검증하는 과정이 필요하다.

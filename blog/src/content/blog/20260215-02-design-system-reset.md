프론트에서 자체 디자인시스템을 오래 굴리다 보면, 문제는 기능보다 경계에서 먼저 터진다. 컴포넌트를 감싸기 위한 래퍼가 늘수록 실제 UI 프레임워크와 팀 코드 사이에 번역층이 생기고, 디자이너·개발자·QA가 같은 화면을 서로 다른 언어로 설명하게 된다. 변경 하나도 스타일 수정, 래퍼 규칙 확인, 테스트 보정으로 쪼개지면서 속도보다 피로가 먼저 올라가는 구조였다.

그래서 논의 끝에 디자인시스템만을 위해 존재하던 래퍼 컴포넌트를 제거하기로 했다. 기획, 디자인, 개발이 같이 합의한 기준은 단순했다. 베이스 UI 프레임워크의 theme과 CSS variable을 중심에 두고, 팀 규칙은 꼭 필요한 범위만 얹는다. 프레임워크가 원래 큰 단위로 해결해 주는 문제를 다시 손으로 쪼개 만들지 말자는 쪽으로 방향을 정리했다.

실행은 새로운 추상화를 추가하는 일이 아니라, 끊어진 연결을 복구하는 작업에 가까웠다. 디자인팀이 작업한 피그마 템플릿에서 플러그인으로 추출한 theme을 MUI에 넣고 그대로 쓰기 시작했다. 토큰이 코드에 바로 매핑되니 디자인 값과 개발 값이 따로 노는 구간이 줄었다. 동시에 프로퍼티를 타입으로 고정해 UI 사용 지점을 더 명확히 했고, 디자인팀도 연결된 컴포넌트 기준으로 커뮤니케이션하기 쉬워졌다.

테스트 원칙도 같은 방향으로 맞췄다. QA에서 testid 가이드가 공유됐지만, testid는 최후의 수단으로 두고 접근성 속성을 먼저 보자는 쪽을 택했다. 이름, 역할, 상태가 드러나는 마크업은 사용자 경험을 올리는 동시에 테스트 기준점도 안정화한다. selector를 임의 문자열에 덜 의존하게 되니 리팩터링 때 테스트가 불필요하게 흔들리는 문제도 줄일 수 있었다.

이 변화가 만능은 아니다. 토큰 네이밍, 컴포넌트 책임, 예외 처리 기준을 계속 맞추지 않으면 래퍼는 다시 늘어난다. 정량 지표는 아직 충분히 정리되지 않아 체감 변화 중심으로만 기록할 수 있지만, 유지보수 비용과 핸드오프 마찰이 줄어든 흐름은 분명했다. 적용 포인트는 크지 않다. 새 레이어를 만들기 전에 “기본 프레임워크와 토큰으로 해결 가능한가”를 먼저 묻고, 접근성을 테스트 전략의 중심으로 두는 것이다.

처음에는 작업계획을 별도 메모장에 두고 대화를 길게 이어가는 방식으로 코딩했다. 짧은 작업은 버텼지만, 복잡해질수록 이전 합의가 위로 묻히고 진행 상태가 틀어졌다. 실제로 현재 상황을 물었을 때 며칠 전에 만들어 두고 갱신하지 않은 md 파일을 근거로 답한 적이 있었다. 그때 대화 컨텍스트만 믿는 방식은 결국 같은 설명을 반복하게 만든다는 걸 확인했다.

그래서 세션 시작 시 저장소 안에 작업 md 파일을 만들고, 커밋할 때마다 완료/진행/남은 작업을 업데이트하게 했다. 코드 작성, 리뷰, 리팩토링, 문서 갱신을 하나의 루프로 묶고, 다음 세션은 대화를 이어붙이는 대신 문서에서 다시 시작했다. 필요하면 컨텍스트를 비워도 기준점이 남으니 흐름이 덜 흔들렸다. 체감상 임시로 쌓는 구조보다 유지 가능한 구조에 가까웠다.

효과가 가장 컸던 구간은 반복 재작성이었다. 구현을 끝낸 뒤 뒤늦게 제약을 발견해 브랜치를 지우고 다시 쓰는 일이 여러 번 생겼다. 이때 작업하면서 파악한 사실과 스펙을 먼저 문서에 기록해 두면 재시작 비용이 낮아진다. 무엇을 왜 버렸는지가 남아 있어서 다시 시작할 때 판단이 빨라지고, 같은 실수를 반복할 가능성도 줄었다.

정리 작업도 AI에게 분리해서 맡겼다. 메인 기준 리베이스, 자잘한 커밋을 의미 단위로 묶기, 임시 브랜치에서 체리픽으로 이력 정리 같은 일들이다. 히스토리가 읽히기 쉬워져 리뷰가 수월해진 건 분명했다. 다만 자동 정리가 항상 안전하다고 보긴 어려워서, 최종 diff와 커밋 메시지는 사람이 다시 확인하는 절차를 고정했다.

지금은 복잡한 작업일수록 작은 단위 구현, 테스트, 문서 갱신, 컨텍스트 리셋을 한 세트로 본다. 핵심은 AI가 모든 맥락을 기억하길 기대하는 게 아니라, 잊어도 복구되는 작업 구조를 먼저 만드는 일이다. 요구사항이 바뀌면 재작업은 여전히 생긴다. 그래도 최신 md 한 장이 살아 있으면 다음 세션에서 어디서부터 다시 들어갈지 빠르게 결정할 수 있다.

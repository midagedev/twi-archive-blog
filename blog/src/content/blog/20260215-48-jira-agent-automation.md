팀 운영에서 시간이 가장 많이 새는 구간은 큰 의사결정이 아니라 반복되는 연결 작업이었다. QA 이슈를 받으면 코드 맥락을 다시 읽고, 팀 보드에 새 이슈를 만들고, 두 이슈를 링크하고, 상태와 담당자를 맞추고, 브랜치를 파는 순서를 매번 손으로 밟았다. 절차는 단순한데 보드 이름, 계정, 브랜치 기준점처럼 작은 조건이 자주 틀어졌다.

그래서 지라 API 몇 개를 통으로 크게 묶기보다, 자주 쓰는 흐름을 먼저 명시하고 단위 스크립트를 조합하는 방식으로 갔다. QA 이슈 하나를 넣으면 코드 분석 뒤에 팀 보드 이슈 생성, 이슈 연결, 진행중 전환, 내 계정 할당, 이슈 번호 기반 브랜치 생성과 체크아웃까지 이어지게 했다. 자동화 범위를 좁게 잡고 팀별 상수를 분리해 변경 비용을 낮추는 데 집중했다.

실제로 까다로운 부분은 API 호출 자체보다 문맥 파라미터였다. QA 보드 이름이 무엇인지, 팀 보드가 어디인지, 어떤 브랜치에서 따야 하는지 같은 정보가 환경마다 달랐다. 이걸 전부 하드코딩하면 유지보수 비용이 금방 커진다. 그래서 흐름 설명은 MD에 짧게 적고, 에이전트가 필요한 단위 스크립트에 채워 넣게 했다. 구현 속도도 빨랐고, 수정 지점이 눈에 잘 보였다.

주간 회의에서는 지라를 붙인 클로드로 팀원별, 에픽별 진행 현황을 실시간 집계해 운영 상태를 같이 봤다. 반응은 엇갈렸다. 편하다는 의견도 있었고, creepy에 가깝다는 반응도 있었다. 여기서 확인한 한계는 분명했다. 자동 집계는 판단을 대신하지 못하고, 각 일이 어떤 에픽에 속하고 우선순위가 어디에 있는지 맞추는 보조 수단으로 쓸 때 효과가 컸다.

이 방식을 적용할 때는 먼저 반복되는 흐름을 작게 고르고, 승인 없이 실행하면 위험한 액션은 분리하고, 회의에서 볼 집계의 목적을 합의해 두는 편이 안전했다. 생산성 개선 폭을 일반화할 수치까지 확보한 것은 아니라서 과장할 수는 없다. 다만 컨텍스트 전환과 누락을 줄이는 데에는 확실히 도움이 됐고, 팀이 같은 언어로 우선순위를 말하는 시간이 늘었다.

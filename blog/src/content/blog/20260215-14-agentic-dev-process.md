처음 시연에서 얻은 핵심은 모델 성능보다 작업 순서였다. 에이전트에게 바로 구현을 맡기면 범위가 커지고 대화 맥락이 금방 흐려졌다. 그래서 시작을 고정했다. 전체 기능 플랜을 먼저 받아 별도로 기록하고, 그중 이번 사이클에서 끝낼 일부 피처만 다시 좁혀 세부 플랜을 받았다. 한 번에 많이 시키지 않고, 기록을 기준으로 범위를 계속 잘라 들어가는 방식이다.

구현 단계는 작은 실패 테스트부터 시작했다. 실패하는 테스트를 만든 뒤 최소 구현으로 통과시키고 바로 커밋한다. 이 단위를 세 번 반복해 최소 기능을 만들고 PR을 연 다음, 다시 다음 목표를 잡아 같은 루프로 돌아갔다. PR을 작게 끊으니 변경 이력이 안정적이었고, 작업자와 모델 모두 이전 대화의 잔여 맥락을 비우고 다시 시작하기 쉬웠다.

이 흐름은 팀 온보딩에도 그대로 적용했다. 새로 합류한 동료와는 먼저 사용 중인 클로드코드 버전과 기본 모델을 최신 상태로 맞추고, 터미널 팬을 나눠 병렬 작업을 진행했다. 작업현황 파악 방식을 확인했을 때는 며칠 전 이후 갱신되지 않은 md 파일을 근거로 삼는 문제도 있었다. 그래서 코드 기준으로 현재 진척을 다시 분석해 문서를 갱신하게 하고, 코드 작성-리뷰-리팩토링-문서 갱신을 같은 단위로 반복하도록 가이드했다.

결론은 단순하다. 에이전트 활용은 감각보다 루프 설계가 먼저다. 플랜 기록, 실패 테스트, 작은 커밋과 PR, 컨텍스트 리셋, 문서 동기화를 한 묶음으로 운영하면 결과가 덜 흔들린다. 다만 정량 지표로 효과를 측정한 단계는 아직 아니고, 문서 갱신이 늦어지면 루프 전체가 쉽게 틀어진다. 적용할 때는 한 피처를 닫는 짧은 사이클부터 시작하고, 다음 프롬프트에 목표와 진척을 함께 적는 규칙을 고정하는 편이 안전했다.

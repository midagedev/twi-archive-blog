2025년 7월, 팀의 기본 도구는 ChatGPT 정도였고 코파일럿이나 에이전트는 없었다. 그래서 도입 발표보다 먼저 한 달 안에 눈에 보이는 결과를 만들기로 했다. 개인 구독 Claude Max로 실제 업무를 처리해 보고, “나도 저렇게 쓰고 싶다”는 관심을 만드는 데 집중했다. 설득 문서보다 반복 작업 감소가 먼저라고 판단했다.

초기 확산은 핸즈온으로 했다. 같은 날 두 세션을 열고 실무 이슈 하나를 잡아 작업계획을 관리하며 작은 단위로 진행했다. 프론트엔드 세션은 잘 맞았지만, 백엔드는 API 통합 테스트를 처음 넣는 구간에서 환경 준비가 막혀 토큰만 크게 쓰고 실패했다. 모델 사용법보다 팀별 선행 환경을 먼저 맞춰야 한다는 걸 여기서 확인했다.

그다음은 귀찮은 절차를 스킬로 고정했다. 저장소 운영이 GitHub 중심이 아니라 PR 생성도 별도 스킬이 필요했고, 그 안에 회사 컨벤션을 넣어 반복 판단을 줄였다. 급할 때는 원격에서 프롬프트만 넘겨 배포까지 이어지게 했다. 핵심은 화려한 자동화가 아니라 자주 틀리는 단계부터 표준화하는 일이었다.

테스트도 같은 원리로 갔다. 미리 만든 테스트 케이스가 쌓이자, 커밋 맥락에서 관련 케이스를 추정해 선택 실행하는 흐름이 생겼다. 코드베이스를 훑어 필요한 테스트를 골라 돌리니 매번 전체를 돌리는 부담이 줄었다. 반면 메신저 연동 이후에는 로컬과 다른 퍼미션, 속도 이슈가 계속 나와 allow tools를 세밀하게 조정해야 첫 PR 생성이 가능했다.

최근에는 컨텍스트를 분리 관리하고 작업 소재를 DB에서 가져오게 한 뒤, 여러 에이전트를 장시간 유지하는 오케스트레이션도 실험했다. compact를 반복하며 몇 시간씩 돌려 토큰 소모가 컸고, 운영 난도도 높았다. 결론은 단순하다. 에이전트 도입은 프롬프트 문제가 아니라 권한, 환경, 테스트, 절차를 함께 다루는 엔지니어링 작업이다. 정량 생산성 수치는 아직 정리 중이다.

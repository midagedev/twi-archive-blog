첫 회사에서 맡은 일은 정부부처와 지자체 사이트에 검색 기능을 붙이는 일이었다. 돌아보면 검색엔진 자체보다 더 자주 막히던 지점은 데이터 출처였다. 어디서 데이터를 가져와야 하는지 담당자가 끝까지 알려주지 못하는 경우가 꽤 있었고, 그때마다 구현보다 먼저 데이터 흐름을 찾아야 했다.

SSH가 열려 있으면 그나마 출발점이 있었다. 소스를 뒤져 데이터 소스를 확인하고, 실제 콘텐츠가 들어 있을 법한 테이블을 추리하는 작업을 반복했다. 문서보다 코드와 스키마에서 단서를 모으는 방식이었다. 검색 기능을 붙이는 일처럼 보여도, 실제로는 서비스 내부 구조를 해석하는 일이 더 큰 비중을 차지했다.

서버 접근이 안 되면 방법을 바꿨다. HTML에서 시작해서 브라우저 요청을 따라가며 정형·비정형 데이터가 어떤 포맷으로 오가는지 확인했다. 거기서 추출 가능한 값을 정리해 메타데이터를 만들었다. 이 과정은 주로 팀장님들이 하는 걸 가까이서 보며 익혔고, 문제를 푸는 순서 자체를 배우는 시간이었다.

이 경험으로 얻은 건 특정 도구 지식보다 패턴을 읽는 습관이었다. 웹서비스와 서버 아키텍처는 겉으로 다 달라 보여도 데이터가 생성되고 전달되는 경로에는 공통점이 있었다. 정보가 불완전한 상황에서는 정답을 기다리기보다, 보이는 단서부터 좁혀가며 구조를 복원하는 방식이 실무에서 더 오래 쓰였다.

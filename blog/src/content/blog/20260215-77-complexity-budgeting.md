기능 수가 늘 때 문제는 선형이 아니라 조합으로 불어난다. 3개면 체감상 1+2+3 정도인데 6개가 되면 순식간에 21처럼 커진다. 어려운 건 코드량보다 경계 수다. 서로 기대는 지점이 많아질수록 테스트 범위, 배포 순서, 롤백 판단이 같이 꼬인다. 그래서 일정이 빡빡할수록 기능 추가보다 먼저 “반드시 같이 움직여야 하는 것”을 줄이는 데 시간을 썼다.

실행은 거창한 재설계보다 경계 정리부터 했다. 조합이 필요한 시그널이나 리시버가 늘어나면 관련 단위를 합치거나 책임을 옮겨 연결 수를 줄였다. 반대로 의존성이 없는 피처는 배포 단위를 분리했다. 효용이 낮은 기능 테스트 때문에 즉시 가치가 나는 기능 배포를 미루지 않기 위해서다. 방법론 이름보다 현재 제약을 우선했고, 객체지향도 목표로 삼기보다 결과로 깔끔해질 때만 택했다.

효과는 의사결정 속도에서 먼저 나왔다. 영향 범위가 좁아지니 PR 리뷰와 테스트 설계가 단순해졌다. 데드코드도 개인 과오로만 보지 않게 됐다. 일정 압박과 우선순위 관리가 무리라는 운영 신호로 보는 편이 더 정확했다. 물론 경계를 줄인다고 항상 정답은 아니다. 도메인 이해가 얕거나 합의가 없으면 다른 결합을 만들 수 있어서, 크게 한 번에 바꾸기보다 작게 바꾸고 관찰했다.

적용 포인트는 단순하다. 독립 피처를 묶어 가치 전달을 늦추고 있지 않은지, 조합 강제가 생긴 경계를 관성으로 두고 있지 않은지, 데드코드를 사람 문제로만 처리하고 있지 않은지를 주기적으로 점검한다. 정량 수치는 팀마다 달라 단정하기 어렵지만, 복잡도를 예산처럼 다루기 시작하면 같은 인원과 비슷한 기능 수에서도 체감 난이도는 내려간다.
